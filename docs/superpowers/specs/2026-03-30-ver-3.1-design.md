## Feature 1: Analytics Audit & Fix

### Root Cause

The Admin Analytics tab reads from Python/SQLite via `/stage-analytics`. The Python `backfill_actuals()` function in `canadian_llm_council_brain.py` is **orphaned — never called from any endpoint or cron**. Meanwhile, the Next.js cron at 4:30 PM correctly fills PostgreSQL with actuals. Two separate databases, no sync. This is why 5D and 8D hit rates never update in the admin panel.

### Fix — Part A: Wire Up Python Backfill

- Add a `POST /backfill-actuals` endpoint on the council FastAPI server (`api_server.py`)
- The endpoint calls `historical_analyzer.backfill_actuals(fetcher)` using the brain's existing `LiveDataFetcher`
- Add a call to this endpoint from the existing 4:30 PM AST cron in `scripts/start-cron.ts`, after the PostgreSQL backfill completes
- This fills the SQLite `accuracy_records` table that `/stage-analytics` reads from
- The Learning Engine's calibration (`build_council_calibration()`) also reads from SQLite — it will start getting real data too

### Fix — Part B: Trust Signals on Analytics UI

Add to every summary card in the Analytics tab (`src/app/admin/page.tsx`):

- `last_updated` timestamp — when the data was last computed
- `sample_size` count — how many picks contribute to each metric
- Amber "Low sample" badge if sample size < 10
- Red "Stale data" badge if `last_updated` > 24 hours old on a trading day (Mon-Fri)

The Python `/stage-analytics` endpoint must return `last_updated` and sample counts alongside existing metrics. Add these fields to the `get_stage_analytics()` return value.

Per-stage rows in the LLM Stage Performance table also get sample counts.

### Files Modified

- `api_server.py` — new `/backfill-actuals` endpoint
- `scripts/start-cron.ts` — add call after PostgreSQL backfill
- `canadian_llm_council_brain.py` — `get_stage_analytics()` returns `last_updated` + sample counts
- `src/app/admin/page.tsx` — trust signals on Analytics tab cards + table
- `src/app/api/admin/analytics/route.ts` — pass through new fields

### Impact on Other Sections

- **Accuracy page** (public): unaffected, reads PostgreSQL directly
- **Learning Engine calibration**: gets correct data (was also starved by the orphaned backfill)
- **XLSX export**: will have real numbers instead of "—"
- **Dashboard/portfolio**: unaffected

### Testing

- Trigger `/backfill-actuals` manually, verify SQLite `accuracy_records` get filled
- Verify `/stage-analytics` returns non-null `hit_rate_5d` and `hit_rate_8d`
- Verify Analytics tab shows updated numbers with timestamps and sample counts
- Verify stale/low-sample badges render correctly (mock scenarios)
- Verify cron triggers both PostgreSQL and SQLite backfills in sequence

## Feature 2: Vertex AI Migration (Stage 2 Gemini)

### Problem

Gemini's public API (`generativelanguage.googleapis.com`) returns frequent 503 errors, causing Stage 2 to skip. This degrades consensus accuracy and starves the Learning Engine's stage weight mechanism (currently 13/30 gate, partly because Stage 2 data is missing).

### Solution

Replace the `google-genai` SDK with `google-cloud-aiplatform` (Vertex AI SDK). Vertex AI uses Google Cloud's regional infrastructure with SLAs.

### Authentication Change

- **Old:** `GOOGLE_API_KEY` (simple API key) → `genai.Client(api_key=...)`
- **New:** Service account JSON file → `vertexai.init(project=..., location=...)`
- New env vars:
  - `GCP_PROJECT_ID=spiketrades`
  - `GCP_LOCATION=northamerica-northeast1` (Montreal — closest to Nova Scotia)
  - `GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account.json`
- Remove `GOOGLE_API_KEY` from `.env` after migration confirmed

### Code Changes in `_call_gemini()`

Replace the SDK imports and client initialization. Keep the function signature identical so `run_stage2_gemini()` doesn't change at all.

Before:
```python
from google import genai
from google.genai import types
client = genai.Client(api_key=api_key)
resp = client.models.generate_content(
    model=model, contents=user_prompt,
    config=types.GenerateContentConfig(
        system_instruction=system_prompt,
        max_output_tokens=max_tokens,
        temperature=temperature,
        response_mime_type="application/json",
    ),
)
return resp.text
```

After:
```python
import vertexai
from vertexai.generative_models import GenerativeModel, GenerationConfig

vertexai.init(project=os.getenv("GCP_PROJECT_ID"), location=os.getenv("GCP_LOCATION", "northamerica-northeast1"))
model_obj = GenerativeModel(model, system_instruction=[system_prompt])
resp = model_obj.generate_content(
    user_prompt,
    generation_config=GenerationConfig(
        max_output_tokens=max_tokens,
        temperature=temperature,
        response_mime_type="application/json",
    ),
)
return resp.text
```

### Fallback Strategy

No fallback to public API. If Vertex AI fails, the existing stage-skip resilience handles it (Stage 2 skips, pipeline continues). We want to see failures clearly, not mask them.

### Requirements

- Replace `google-genai>=1.0.0` with `google-cloud-aiplatform>=1.50.0` in `requirements-council.txt`

### GCP Setup Required (User)

1. Enable Vertex AI API in the `spiketrades` GCP project
2. Create a service account with "Vertex AI User" role
3. Download the JSON key file
4. Deploy key file to server and set `GOOGLE_APPLICATION_CREDENTIALS` env var

### Files Modified

- `canadian_llm_council_brain.py` — `_call_gemini()` function rewritten
- `requirements-council.txt` — SDK swap
- `.env.example` — new GCP env vars, remove `GOOGLE_API_KEY`
- `.env` on server — new credentials
- Docker/deployment — service account JSON file deployed

### Impact on Other Sections

- **Learning Engine**: Stage 2 weights mechanism gets consistent data, should reach 30-pick gate faster
- **Admin Council tab**: Data Source Health won't show Gemini 503 errors (they should stop)
- **Run Status** (Feature 3): Stage 2 will show as succeeding instead of "SKIPPED"
- **Analytics**: hit rates improve over time as full 4-stage council produces better consensus

### Testing

- Call `_call_gemini()` directly with a test prompt, verify response parses as JSON
- Run a full council pipeline, verify Stage 2 completes without 503
- Verify retry logic still works (simulate transient error)
- Verify stage-skip still works if Vertex AI is misconfigured (remove credentials, confirm graceful skip)
- Check `/run-status` shows Stage 2 as "complete" not "skipped"

## Feature 3: Live Run Status

### Current State

The Council tab shows a simple green "Idle" / amber "Running" dot with elapsed time, polled every 30 seconds. No visibility into stages, batches, or skip events.

### Design — Log-Based Progress Tracking

#### Python Side (council FastAPI)

Add a shared `run_progress` dict in `api_server.py` that gets updated as the council brain runs.

Progress structure:
```json
{
  "running": true,
  "trigger": "scheduled",
  "started_at": "2026-03-31T10:45:02",
  "current_stage": "stage2_gemini",
  "stages": {
    "pre_filter": { "status": "complete", "tickers_in": 347, "tickers_out": 142, "duration_s": 45 },
    "stage1_sonnet": { "status": "complete", "batches_done": 7, "batches_total": 7, "picks": 48, "duration_s": 380 },
    "stage2_gemini": { "status": "running", "batches_done": 3, "batches_total": 5, "picks": null, "duration_s": 124 },
    "stage3_opus": { "status": "pending" },
    "stage4_grok": { "status": "pending" },
    "consensus": { "status": "pending" }
  },
  "skipped_stages": [],
  "elapsed_s": 504,
  "last_completed_run": {
    "trigger": "scheduled",
    "completed_at": "2026-03-30T11:23:45",
    "total_duration_s": 1073,
    "picks": 10,
    "skipped_stages": ["stage2_gemini"],
    "stages_summary": {
      "pre_filter": { "status": "complete", "tickers_in": 347, "tickers_out": 142, "duration_s": 45 },
      "stage1_sonnet": { "status": "complete", "picks": 48, "duration_s": 380 },
      "stage2_gemini": { "status": "skipped", "reason": "503 timeout after 420s", "duration_s": 420 },
      "stage3_opus": { "status": "complete", "picks": 12, "duration_s": 195 },
      "stage4_grok": { "status": "complete", "picks": 10, "duration_s": 88 },
      "consensus": { "status": "complete", "picks": 10, "duration_s": 3 }
    }
  }
}
```

Implementation approach:
- Add a `ProgressTracker` class that wraps the shared `run_progress` dict with methods like `start_stage()`, `update_batch()`, `complete_stage()`, `skip_stage(reason)`
- Pass the tracker into `run_council()` as an optional parameter
- Add explicit `tracker.start_stage("stage1_sonnet")` / `tracker.complete_stage(picks=48)` calls at each stage boundary inside `run_council()`
- This is more reliable than parsing log strings — explicit calls at known points in the pipeline
- The `trigger` field is set to "scheduled" or "manual" based on how the run was initiated

New endpoint: `GET /run-status` returns the current `run_progress` dict.

When idle, `last_completed_run` shows the final status of the most recent run.

#### Next.js Side (Admin Council Tab)

Replace the simple Running/Idle indicator with a **stage pipeline visualization**:

- Horizontal step display: Pre-filter → Sonnet → Gemini → Opus → Grok → Consensus
- Active stage: amber pulsing dot + batch progress ("3/5 batches")
- Completed stage: green checkmark + pick count + duration
- Skipped stage: red X + skip reason
- Pending stage: dim/grey
- Total elapsed time displayed
- Trigger label: "Scheduled run (10:45 AM)" or "Manual run (triggered by admin)"
- Poll interval: 10 seconds (down from 30s)
- When idle: show `last_completed_run` summary

Positioned within the existing Manual Scan area of the Council tab.

#### Next.js API Route

- New route `GET /api/admin/council/status` (or extend existing `/api/admin/council` GET)
- Proxies to Python `/run-status`
- Auth required (admin only)

### Files Modified

- `api_server.py` — `run_progress` dict, log handler, `/run-status` endpoint, trigger tracking
- `src/app/admin/page.tsx` — stage pipeline visualization replacing simple indicator
- `src/app/api/admin/council/route.ts` — proxy `/run-status` or extend existing GET

### Impact on Other Sections

- **Council brain**: no changes to core logic — we read logs, don't modify the pipeline
- **FMP Data Source Health**: stays where it is, below the status area
- **Manual scan trigger**: still works as before, just with richer status feedback
- **Existing health endpoint**: unchanged, `/run-status` is additive

### Testing

- Trigger a manual run, verify `/run-status` updates through all stages
- Verify stage transitions appear correctly (pending → running → complete)
- Simulate a stage skip (e.g., temporarily break Vertex AI auth), verify "skipped" status + reason
- Verify idle state shows `last_completed_run` summary
- Verify 10-second polling doesn't create excessive load
- Verify trigger label shows "manual" for admin-triggered and "scheduled" for cron

## Feature 4: Dividend Display & Warning

### Data Source

FMP `/stable/stock-dividend-calendar` endpoint — returns ex-dividend date, payment date, dividend amount, and yield per ticker.

### New FMP Function

Add `getDividendInfo(ticker: string)` to `src/lib/api/fmp.ts`:
- Calls FMP dividend calendar endpoint for the specific ticker
- Returns: `{ exDate, paymentDate, amount, yield, frequency }` or `null` if no upcoming dividend
- Results cached per ticker per day (dividend calendars are stable intraday)

### Analysis Page Header — Dividend Badge

On the existing header line with the ticker name and sector category bubble (`src/app/dashboard/analysis/[id]/page.tsx`):

- Add a **"DIV" pill badge** after the sector bubble, only for dividend-paying stocks
- Style: distinct color (gold/amber), same shape as sector bubble
- Content: `DIV $0.52 | Ex: Apr 3`
- Only renders if `getDividendInfo()` returns a non-null result with a future or recent ex-date

### Ex-Dividend Warning Banner

Below the 3/5/8-day target cards:

- Amber warning banner, only shown when ex-date falls within the prediction window (report date + 1 to report date + 8 trading days)
- Text: "Ex-dividend date (Apr 3) falls within the prediction window. Stock price typically drops by approximately the dividend amount ($0.52 / ~1.2%) on this date."
- Percentage calculated as: `(dividend_amount / current_price) * 100`

### What This Does NOT Do

- Does not modify the council brain scoring or LLM prompts
- Does not add dividend data to `StockDataPayload` or any Pydantic model
- Does not affect dashboard spike cards (analysis detail page only)
- Does not create a dedicated dividend section
- Does not store dividend data in Prisma — fetched live from FMP

### Files Modified

- `src/lib/api/fmp.ts` — new `getDividendInfo()` function
- `src/app/api/spikes/[id]/route.ts` — add `dividend` field to response
- `src/app/dashboard/analysis/[id]/page.tsx` — dividend badge + warning banner

### Impact on Other Sections

- **Spike detail API**: new optional `dividend` field in response — existing consumers ignore unknown fields
- **Dashboard spike cards**: unaffected
- **Council brain**: unaffected
- **Portfolio page**: unaffected
- **If FMP endpoint fails or returns empty**: badge simply doesn't render, no error state

### Testing

- Verify `getDividendInfo()` returns data for known dividend stocks (e.g., RY.TO, TD.TO, BCE.TO)
- Verify null return for non-dividend stocks (e.g., SHOP.TO)
- Verify badge renders on analysis page for dividend stocks
- Verify badge does not render for non-dividend stocks
- Verify warning banner appears when ex-date is within prediction window
- Verify warning banner does not appear when ex-date is outside window
- Verify percentage calculation is correct
- Verify graceful handling when FMP endpoint returns error/empty
