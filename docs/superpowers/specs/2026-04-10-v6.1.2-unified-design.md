### Problem

- `runEveningScan()` (lines 478-804, 327 lines)
- `runMorningScan()` (lines 816-1260, 410 lines)

### Solution
Extract 6 shared helpers (5 existing + 1 new vault). Reduce to 2 thin wrappers. Delete `runDailyAnalysis()`.

### Shared Helpers

#### `callCouncilBrain(opts: CouncilCallOpts): Promise<CouncilMappedResponse>`
- **Params:** `{ trigger: 'morning' | 'evening', cached?: boolean }`
- **Logic:**
  - If `cached`: GET `${COUNCIL_API_URL}/latest-output-mapped` (60s timeout)
  - If live: POST `${COUNCIL_API_URL}/run-council-mapped?trigger=${trigger}` (3600s timeout)
- **Returns:** Parsed `CouncilMappedResponse`
- **Error handling:** Throws with descriptive message on timeout/parse failure
- **Source lines:** Currently duplicated at 104-176, 503-541, 838-912

#### `writeScanArchive(scanType: 'EVENING' | 'MORNING', data: ArchiveData): Promise<string>`
- **Params:** `{ scanDate, scanGeneratedAt, runId, regime, macroContext, topPicks, councilLog, riskSummary, dailyRoadmap, rawCouncilOutput, universeSize, tickersScreened, skippedStages }`
- **Logic:**
  - Writes to `EveningScanArchive` or `MorningScanArchive` based on scanType
  - Archive write must succeed or scan aborts (archive-first safety pattern)
- **Returns:** Archive row ID
- **Source lines:** Currently duplicated at 560-591, 931-962

#### `buildSpikeData(councilSpikes: any[], reportId: string, scanType: ScanType): Prisma.SpikeCreateInput[]`
- **Params:** Raw council spike data, report ID, scan type
- **Logic:** Maps 50+ fields from council output to Prisma Spike model
  - Core: rank, ticker, name, sector, exchange, price, volume, avgVolume, marketCap
  - Scores: spikeScore, momentumScore, volumeScore, technicalScore, macroScore, sentimentScore
  - Creative factors: shortInterest, volatilityAdj, sectorRotation, patternMatch, liquidityDepth, insiderSignal, gapPotential
  - Predictions: predicted3Day, predicted5Day, predicted8Day, confidence
  - Technicals: rsi, macd, macdSignal, adx, bollingerUpper, bollingerLower, ema3, ema8, atr
  - Hit Rate 2.0: setupRateCILow/High, tickerRate, tickerRateSamples, tickerRateCILow/High, tickerMedianMoveOnHits/Misses, setupMedianMoveOnHits/Misses, setupRateRegime, calibrationReconciliation
  - IIC: institutionalConvictionScore
  - Learning: learningAdjustments
  - v6.1.0: scanType, scanGeneratedAt, scanReferencePrice
- **Source lines:** Currently duplicated at 224-269, 629-676, 1000-1047

#### `saveScanReport(opts: SaveReportOpts): Promise<{ reportId: string, spikeCount: number }>`
- **Params:** `{ date, scanType, councilData, spikeData }`
- **Logic:**
  1. Check for existing report with same date+scanType
  2. If re-running: delete old PortfolioEntry refs + Spike rows (scoped DELETE with WHERE)
  3. Upsert DailyReport row with market data (tsx, oil, gold, btc, cadUsd, marketRegime)
  4. Create Spike rows via buildSpikeData()
  5. Save CouncilLog (analysis, consensus score, processing time)
- **Returns:** Report ID and spike count
- **Source lines:** Currently duplicated at 189-306, 600-711, 971-1091

#### `sendScanEmail(scanType: 'EVENING' | 'MORNING', opts: EmailOpts): Promise<void>`
- **Params:** `{ date, spikes, councilData, isDegraded, skippedStages }`
- **Logic:**
  - **Degraded-run gate:** If any LLM stages skipped/failed:
    - Admin-only notification with warning prefix
    - User-facing emails suppressed
  - **MORNING path:**
    - Try rich HTML email from Python `/render-email` endpoint (120s timeout)
    - Fallback to simple `sendDailySummary()` if HTML render fails
    - Send to all users with `emailDailySpikes=true`
  - **EVENING path:**
    - Send evening preview email via `sendEveningPreviewEmail()`
    - Send to all users with `emailEveningPreview=true`
- **Source lines:** Currently duplicated at 308-457, 724-782, 1096-1239

#### `writeVaultSnapshot(scanType: 'EVENING' | 'MORNING', data: VaultData): Promise<void>` -- NEW
- **Params:** `{ scanDate, archiveRow, report, spikes, councilLog }`
- **Logic:** (async, fire-and-forget -- scan success is never affected)
  1. Build snapshot JSON containing:
     - meta: scanType, scanDate, generatedAt, version
     - archive: full archive row (complete council output)
     - report: DailyReport row with market data
     - spikes: all 10 Spike rows with full scoring
     - councilLog: stage metadata, consensus, timing
     - accuracyDelta: accuracy_records updated since last snapshot
     - userPortfolioDelta: new PortfolioEntry rows since last snapshot
  2. Gzip compress (~50-100KB per snapshot)
  3. Write to `/opt/spike-trades/spiketrades-vault/vault/{date}/{scanType}-{timestamp}.json.gz`
  4. Git add + commit + push to `spiketrades-vault` private repo
  5. If push fails: log warning, local copy preserved, next scan will retry
- **Git operations:** Uses `child_process.execSync` for git add/commit/push in the vault repo clone at `/opt/spike-trades/spiketrades-vault/`. Falls back gracefully if git is unavailable or repo not cloned (dev environment).
- **Error handling:** Wrapped in try/catch. Failures logged but never propagate to scan result.

## 2. Weekend Scan Schedule

### Problem
Evening scan runs Mon-Fri at 8 PM ADT. Friday and Saturday have no evening scan, so there are no Tomorrow's Spikes for Monday. Sunday evening should produce Monday's picks using Friday EOD data + weekend news.

### Solution

#### 2a. Cron Expression
**File:** `scripts/start-cron.ts` line 72

| | Before | After |
|---|---|---|
| Expression | `0 20 * * 1-5` | `0 20 * * 0-4` |
| Days | Mon-Fri | Sun-Thu |
| Meaning | 8 PM on trading days | 8 PM on evenings before trading days |

#### 2b. New Guard Function
**File:** `src/lib/utils/index.ts` (or co-located)

```typescript
/**
 * Returns true if today is a valid day for evening scans.
 * Evening scans run Sun-Thu to produce picks for the NEXT trading day.
 * Sunday's scan produces Monday's picks using Friday EOD + weekend news.
 */
export function isEveningScanDay(date: Date = new Date()): boolean {
  const day = date.getDay(); // 0=Sun, 1=Mon, ..., 4=Thu
  return day >= 0 && day <= 4;
}
```

- Does NOT check TSX holidays (the evening scan before a holiday still runs -- the morning scan's `isTradingDay()` guard handles holiday skips)
- `isTradingDay()` remains unchanged for morning scan (Mon-Fri, excludes holidays)

#### 2c. Route Guard Update
**File:** `src/app/api/cron/scan-evening/route.ts`
- Replace `isTradingDay()` call with `isEveningScanDay()`

#### 2d. Sunday Date Logic
**No changes needed.** The evening scan computes `getNextTradingDay()` for the report date:
- Sunday evening: next trading day = Monday -> report date = Monday (correct)
- Thursday evening: next trading day = Friday -> report date = Friday (correct)
- Friday: no scan (not in Sun-Thu cron)
- Saturday: no scan (not in Sun-Thu cron)

#### 2e. Council Brain Behavior on Sunday
**No changes needed.** The Python council brain already:
- Fetches the most recent trading day's OHLCV data (Friday's closes on Sunday)
- Fetches fresh news/catalysts from APIs (includes Saturday + Sunday news)
- This is the same behavior as a Monday morning scan, just shifted to Sunday evening

### Edge Cases
- **Holiday Monday:** Sunday evening scan runs, produces picks for Monday. Monday morning scan's `isTradingDay()` skips the morning run. Tomorrow's Spikes page shows Monday picks but no Today's Spikes. This is acceptable -- the evening picks provide coverage.
- **Holiday Friday:** Thursday evening scan runs for Friday. Friday morning's `isTradingDay()` skips. Sunday evening scan runs for Monday using Thursday EOD data (last trading day). Correct.

## 3. Scan-Type Separation (Dual-Scan Data Integrity)

### Problem
The v6.1.0 two-scan architecture added scanType ('MORNING' | 'EVENING') to DailyReport and Spike tables, and the viewer pages (Today's/Tomorrow's Spikes, Archives) correctly filter by scanType. However, the analytics, accuracy, admin, and tracking infrastructure was never updated. This causes:

1. **Schema blockers** -- CouncilLog, AccuracyRecord, and MarketRegime have unique constraints on `date` only, blocking or corrupting the second scan per day
2. **Data mixing** -- Admin panel, accuracy system, and Python historical analyzer pool morning and evening data together
3. **Regime contamination** -- Morning scores (intraday data) and evening scores (EOD data) are fundamentally different prediction regimes but are aggregated as one population

## 4. Data Vault

### Problem
Current backup situation has three vulnerabilities:
1. **Single server** -- all backups (daily pg_dump + SQLite) live on the same DigitalOcean droplet
2. **20-hour blind spot** -- daily backup at 4 AM, scans at 11:15 AM and 8 PM = up to 20h of unprotected data
3. **No offsite copy** -- if the droplet dies, all data is lost

### Solution
Post-scan vault snapshots pushed to a private GitHub repo (`spiketrades-vault`), plus weekly full backups.

### Architecture

```
┌─ Scan completes ─────────────────────────────────────────┐
│ runEveningScan() / runMorningScan()                       │
│   1-6. existing scan steps                               │
│   7. writeVaultSnapshot()   ← async fire-and-forget      │
└──────────────────────────────┬───────────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │  Local vault dir    │
                    │  /opt/spike-trades/ │
                    │  spiketrades-vault/ │
                    │  vault/2026-04-10/  │
                    └──────────┬──────────┘
                               │ git commit + push
                    ┌──────────▼──────────┐
                    │  GitHub (private)    │
                    │  spiketrades-vault   │
                    │  offsite backup      │
                    └─────────────────────┘
```

### Per-Scan Snapshot Contents (~50-100KB compressed)

```json
{
  "meta": {
    "scanType": "MORNING",
    "scanDate": "2026-04-10",
    "generatedAt": "2026-04-10T15:15:32Z",
    "version": "6.1.2"
  },
  "archive": { "full EveningScanArchive or MorningScanArchive row" },
  "report": { "DailyReport row with all market data fields" },
  "spikes": [ "all 10 Spike rows with full scoring" ],
  "councilLog": { "stage metadata, consensus, timing" },
  "accuracyDelta": [ "accuracy_records updated since last snapshot" ],
  "userPortfolioDelta": [ "new PortfolioEntry rows since last snapshot" ]
}
```

### Vault Repo Structure

```
spiketrades-vault/           (private GitHub repo)
├── vault/
│   ├── 2026-04-10/
│   │   ├── morning-2026-04-10T15-15-32Z.json.gz
│   │   └── evening-2026-04-10T00-00-12Z.json.gz
│   ├── 2026-04-11/
│   │   └── ...
├── full-backup/
│   ├── weekly-2026-04-06-postgres.sql.gz
│   ├── weekly-2026-04-06-council.db
│   └── weekly-2026-04-13-postgres.sql.gz
├── scripts/
│   └── restore.ts           (restore utility)
└── README.md
```

### Weekly Full Backup (automated)
Enhanced `spike_trades_daily_backup.sh`:
- Every day: existing pg_dump + SQLite backup to local `/opt/spike-trades/backups/` (unchanged)
- **Sundays only**: also copy pg_dump + SQLite to the vault repo under `full-backup/` and push to GitHub
- Day-of-week check: `if [ "$(date +%u)" = "7" ]; then ...`
- Retention: 30 days local (existing), unlimited on GitHub (git history)

### Full Sunday Automation Sequence
```
4:00 AM AST  — Calibration refresh (existing cron, automatic)
4:05 AM AST  — Daily backup (existing host cron, automatic)
               └── Sunday: also pushes full backup to vault repo (NEW)
8:00 PM AST  — Sunday evening scan (NEW weekend schedule)
               └── writeVaultSnapshot() pushes per-scan snapshot to vault (NEW)
```

### Failure Handling
1. Scan completes successfully -> scan result returned to caller
2. Vault snapshot starts (async, non-blocking)
3. Write local JSON.gz -> always succeeds (local disk)
4. Git commit + push to vault repo -> may fail (network)
5. If push fails:
   - Log warning: "Vault push failed, local copy preserved"
   - Next scan's snapshot will include the missed data
   - Daily backup script can sweep unpushed local snapshots
6. Scan success is NEVER affected by vault status

### Restore Workflows

**Path A -- Single scan corruption** (e.g., Spike rows got mangled):
```bash
tsx scripts/restore.ts vault/2026-04-10/morning-2026-04-10T15-15-32Z.json.gz
# Restores: DailyReport, Spikes, CouncilLog from snapshot
```

**Path B -- Full database loss** (catastrophic):
```bash
# 1. Restore from latest weekly full backup
gunzip full-backup/weekly-2026-04-06-postgres.sql.gz
psql -U spiketrades spiketrades < weekly-2026-04-06-postgres.sql
# 2. Replay per-scan snapshots after the weekly backup date
tsx scripts/restore.ts vault/2026-04-07/ vault/2026-04-08/ vault/2026-04-09/ vault/2026-04-10/
```

### What This Solves

| Vulnerability | Before | After |
|--------------|--------|-------|
| Single server | All backups on same droplet | Offsite copy on GitHub |
| 20-hour blind spot | Daily backup at 4 AM only | Snapshot after every scan |
| No disaster recovery | Manual intervention | `restore.ts` for partial + full restore |
| Data audit trail | Only archive tables | Full timestamped snapshots with deltas |

## Testing Strategy

### Machine A -- Before Push
1. `npm run build` -- verify no TypeScript errors after refactor
2. Verify all imports of `runDailyAnalysis` are removed
3. Verify `runMorningScan` and `runEveningScan` exports are unchanged (API contract preserved)
4. Verify `isEveningScanDay()` returns correct values:
   - Sunday (0) -> true
   - Monday (1) -> true
   - Thursday (4) -> true
   - Friday (5) -> false
   - Saturday (6) -> false
5. Verify schema changes: CouncilLog, AccuracyRecord, MarketRegime all have scanType field + updated unique constraints
6. Verify `/api/accuracy` accepts and filters by scanType parameter
7. Verify `/api/accuracy/check` runs backfill loop for both MORNING and EVENING
8. Verify admin council route includes scanType in Recent Reports response
9. Verify Python pick_history and accuracy_records tables have scan_type column
10. Verify `writeVaultSnapshot()` handles errors gracefully (doesn't throw)
11. Verify vault snapshot JSON structure includes all required fields

### Machine B -- After Deploy
1. All 6 containers healthy
2. `prisma db push` completes without errors (schema changes applied)
3. Both scan pages load with current data
4. Calibration base rates populated (Setup Rate bars showing on SpikeCards)
5. Admin panel: Recent Reports shows scanType badges (Today's / Tomorrow's)
6. Admin panel: Analytics tab has Morning/Evening toggle
7. Accuracy page has Morning/Evening tabs
8. Vault repo exists on GitHub with deploy key configured
9. Local backup files exist and are non-empty
10. `git tag -l` shows only `v6.1.2`
11. No stale issues/PRs/branches on GitHub
